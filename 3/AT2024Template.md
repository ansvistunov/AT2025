# 1. Введение
Данный документ содержит спецификацию требований к программному обеспечению (SRS) для автоматизированной системы управления заказами в розничном магазине (АСУ ЗРМ). Документ определяет функциональные и нефункциональные требования к системе, а также описывает основные характеристики и ограничения разрабатываемого ПО.

## Цели
Цель настоящего документа – формализовать требования к разработке программного обеспечения АСУ ЗРМ.\
Документ предназначен для использования:
   - Заказчиком, для проверки соответствия функциональности системы бизнес-требованиям;
   - Разработчиком, для проектирования, реализации и тестирования системы;
   - Тестировщиком, для построения планов и сценариев тестирования;
   - Сопровождающим персоналом, для эксплуатации и дальнейшего развития системы.

Назначение документа – обеспечить однозначное понимание всех требований к системе, включая функциональные, нефункциональные, интерфейсные и эксплуатационные характеристики, а также определить ограничения и предположения, влияющие на разработку и внедрение.

## Границы применения
Автоматизированная система управления запасами для розничного магазина (АСУ ЗРМ) предназначена для автоматизации процессов учета, контроля и анализа товарных запасов.\
Программное обеспечение охватывает следующие функциональные области:
   - учет поступлений товаров от поставщиков;
   - учет продаж и автоматическое списание товаров;
   - контроль остатков и уведомление о достижении пороговых значений;
   - формирование аналитических отчетов по движению товаров;
   - поддержка интеграции с кассовыми терминалами и системой бухгалтерского учета.

АСУ ЗРМ не включает в себя:
   - функции финансового учета вне контекста товарных операций;
   - управление персоналом;
   - CRM-функциональность.

Система предназначена для эксплуатации в розничных торговых точках малого и среднего масштаба, функционирующих в локальной сети или облачной инфраструктуре.

## Термины и аббревиатуры
- **АСУ ЗРМ** – автоматизированная система управления запасами для розничного магазина.
- **SKU - Stock Keeping Unit** – уникальный идентификатор товарной позиции.
- **API - Application Programming Interface** – интерфейс взаимодействия программных компонентов.
- **GUI	- Graphical User Interface** – графический пользовательский интерфейс.
- **БД** – база данных.
- **Пользователь** – сотрудник магазина, взаимодействующий с системой.
- **Z-отчет** – электронный документ кассовой техники, который формируется по окончании кассовой смены и содержит информацию 
  о продаже товаров за это время.
- **НСИ** – нормативно-справочная информация.
- **Приходные накладные** – электронный учетный документ установленной формы, фиксирующий факт поступления товаров от поставщика.

# 2. Общее описание

## Описание изделия
- **Интерфейсы системы**: Система предназначена для автоматического формирования заказов товаров, состоящая из следующих компонентов:
   - Приложения, установленного локально на рабочей станции.
   - Встроенной базы данных для хранения нормативно-справочной информации (статистика продаж и поступлений товаров, каталог поставщиков с расценками и сроками поставки, неснижаемые остатки).
   - Механизма обмена файлами с кассовым аппаратом и поставщиками.

- **Интерфейсы пользователя**:
   - **Оператор НСИ**: интерфейс локального приложения, реализованный в виде вкладок и таблиц. Предусмотрены формы для ввода и редактирования нормативно-справочной информации. Интерфейс содержит меню навигации, панели инструментов и окна уведомлений об успешных операциях или ошибках.
   - **Владелец магазина**: интерфейс для просмотра отчетов по остаткам и ожидаемым поставкам с возможностью фильтрации по товарам и датам.
   - **Администратор**: интерфейс для назначения ролей другим пользователям.

- **Интерфейсы аппаратных средств ЭВМ**:
   - Персональный компьютер под управлением Windows.
   - Кассовое оборудование, обеспечивающее выгрузку Z-отчетов.
   - Почтовое подключение для отправки заказов поставщикам.

- **Интерфейсы программного обеспечения**:
   - Операционная система: Windows.
   - СУБД: локальная реляционная база данных (SQLite или аналог).
   - Отправка заказов через почтовый сервер поставщикам по протоколу SMTP.

- **Интерфейсы коммуникаций**:
   - Локальная файловая система или внутренняя сеть.

- **Ограничения памяти**: Не предъявляются особые требования.

- **Действия**:
   - Загрузка пользователем Z-отчетов, приходных накладных и расценок поставщиков.
   - Редактирование нормативно-справочной информации.

## Функции изделия
- Автоматический мониторинг запасов товаров.
- Формирование заказов при достижении неснижаемого уровня.
- Автоматическая рассылка заказов поставщикам.
- Учет цен и условий поставки.
- Выбор поставщика с учетом цены и сроков доставки.
- Обеспечение работы системы в автономном режиме, с автоматическим восстановлением после сбоев.
- Возможность загрузки данных вручную через интерфейс.
- Генерация отчетов для владельца магазина.

## Характеристики пользователей
- **Администратор**:
   - IT-специалист с глубокими знаниями и навыками администрирования.
   - Выполняет функции: настройка параметров системы, а также управление ролями, настройками нормативно-справочной информации, загрузкой прайс-листов.
- **Оператор НСИ**:
   - Обладает базовыми навыками работы с ЭВМ.
   - Выполняет функции: ввод и обновление данных о поставщиках, прайсах.
- **Руководитель/владелец магазина**:
   - Обладает базовыми навыками работы с ЭВМ.
   - Выполняет функции: просмотр текущих остатков, планов поставок, отчетов.

## Ограничения
- Система функционирует локально, без подключения к внешним веб-сервисам.
- Обмен данными выполняется через локальные файлы и внутреннюю базу данных.
- Работа возможна на персональном компьютере под управлением Windows.
- Аутентификация пользователей осуществляется по логину и паролю.
- После сбоя система должна восстанавливаться в течение 4 часов.
- Работа допускается в изолированной сети, не требующей высокой пропускной способности.

## Предположения и зависимости
- Данные с кассы и приходных накладных поступают регулярно и в стандартизированном формате.
- Поставщики предоставляют расценки и даты поставок, которые система использует для оптимизации заказов.
- Вся необходимая нормативно-справочная информация предоставляется и обновляется через интерфейс загрузки данных.
- Вся логика работы системы — автоматическая, без необходимости вмешательства оператора после загрузки данных.
- Время формирования заказа — не более 30 минут после запуска системы утром.
- В системе реализована многопользовательская аутентификация по логину и паролю с разграничением ролей.
