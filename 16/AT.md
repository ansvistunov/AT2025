# 1. Введение

## Цели
В этом документе представлена спецификая требований к программному обеспечению для системы обучения персонала(СОП). Он предназначен для всех сотрудников компании. В нём описаны область применения системы, функциональные и нефункциональные требования к программному обеспечению, ограничения при проектировании и системные интерфейсы.

## Границы применения
Система обучения персонала предназначена для развития квалификации сотрудников с помощью знаний других сотрудников (внутренний обмен опытом). Она обеспечивает основу для:

Формирования и регистрации потребностей в обучении (заявки).

Планирования и бронирования внутренних обучающих мероприятий (лекций, семинаров).

Учета и оценки проведенного обучения (отзывы и оценки усвоения).

Управления календарем обязательных тренингов (пожарная безопасность, техника безопасности, вводные программы)

Организации заказа тренингов у внешних провайдеров (сбор предложений, выбор программ)

Система не является платформой для размещения готовых курсов или внешней системой управления обучением (LMS).
## Термины и аббревиатуры
СОП	Система Обучения Персонала.

SRS	Software Requirements Specification (Спецификация требований к программному обеспечению).

Тимлид	Руководитель команды, инициирующий заявку на обучение для своих сотрудников (Заказчик обучения).

Преподаватель	Сотрудник, откликающийся на заявку и проводящий обучение (Исполнитель).

Обучаемый	Сотрудник, посещающий занятие и оставляющий отзыв.

LMS Внешняя система управления обучением.

Координатор обучения Сотрудник, ответственный за создание и ведение календаря обязательных тренингов

Менеджер по внешним тренингам Сотрудник, работающий с внешними тренинг-центрами

Обязательный тренинг Обучение, которое является обязательным для определенных категорий сотрудников согласно требованиям законодательства, отраслевых правил или внутренних политик компании

Внешний тренинг Обучение, которое проводится сторонними специалистами или организациями за пределами компании

Раздел 1 (Введение) предоставляет общий обзор описываемого программного обеспечения. Рекомендуется ознакомиться с этим разделом, поскольку он помогает понять основные принципы, цели данного документа и назначение СОП. Здесь также приведены важные термины и определения.

# 2. Общее описание

## Описание изделия
- **Интерфейсы системы**: Веб-интерфейс, доступный через браузер. REST API для потенциальной будущей интеграции с корпоративными системами.
- **Интерфейсы пользователя**: Адаптивный веб-интерфейс с отдельными представлениями для Тимлидов, Преподавателей и Обучаемых. Включает дашборды, формы создания заявок, каталог мероприятий и систему отзывов.
- **Интерфейсы аппаратных средств ЭВМ**: Не требуются. Система является веб-приложением, развертываемым на серверах компании. Пользователи взаимодействуют с системой с помощью стандартных клиентских устройств (ПК, ноутбуки).
- **Интерфейсы программного обеспечения**: Интеграция с корпоративной службой каталогов (Active Directory/LDAP) для аутентификации. Серверная часть использует СУБД (например, PostgreSQL) для хранения данных.
- **Интерфейсы коммуникаций**: Все взаимодействия с системой должны осуществляться по защищенному протоколу HTTPS.
- **Ограничения памяти**: Система должна быть рассчитана на хранение данных о всех сотрудниках компании, истории заявок, мероприятий и отзывов за несколько лет. Требования к дисковому пространству должны быть определены исходя из расчетного количества пользователей и активности.
- **Действия**: Система обрабатывает следующие основные операции: создание/редактирование заявок на обучение, отклик на заявки с предложением времени, бронирование мероприятия, запись сотрудников на мероприятие, расчет минимального количества участников, сбор и хранение отзывов и оценок.

## Функции изделия
1. Управление заявками на обучение (создание, просмотр, редактирование)

2. Система откликов преподавателей с предложением временных слотов

3. Бронирование мероприятий и планирование расписания

4. Управление квотами участников:

	- Установка максимального количества участников преподавателем

	- Автоматический контроль минимального количества участников системой

5. Регистрация сотрудников на мероприятия

6. Система отзывов и оценок:

	- Отзывы обучаемых о качестве занятий

	- Оценки преподавателей об усвоении материала

7. Назначение конкретных сотрудников на мероприятия тимлидами

8. Управление календарем обязательных тренингов:

	- Создание и планирование событий

	- Привязка к событиям

9. Автоматическая запись сотрудников на обязательные тренинги

10. Организация заказов на внешние тренинги

11. Сопоставление и выбор предложений от внешних организаций

## Ограничения
1. Регистрация пользователей только через корпоративную систему (Active Directory).

2. Отсутствие глобального ограничения на максимальное количество участников.

3. Минимальное количество участников устанавливается системой и не может быть изменено пользователями.

4. Отсутствие интеграции с внешними календарями на первом этапе.

5. Система не поддерживает самостоятельную регистрацию пользователей.

## Предположения и зависимости
1. Все пользователи имеют стабильный доступ к корпоративной сети и веб-браузеру

2. Система зависит от доступности корпоративной службы каталогов для аутентификации

3. Пользователи добросовестно используют систему (своевременно вносят данные, не злоупотребляют отменами)

4. Функциональность email-уведомлений не является обязательной для базовой версии

5. Система развертывается на стандартных серверах без требования специального оборудования

# 3. Детальные требования
Этот раздел детализирует требования к системе, включая функциональные и нефункциональные аспекты, необходимые для успешной реализации проекта.

Перед перечислением функциональных требований ниже приведено описание атрибутов (свойств), используемых для каждого требования.

У каждого функционального требования есть следующие атрибуты:

* **Идентификатор**
    * Уникальный код, который идентифицирует требование во всей документации проекта. Он позволяет ссылаться на конкретное требование без двусмысленности.
    * Структура: `FR-XX-NN`, где `FR` = Функциональное требование, `XX` = код роли (TL, TR, HR и т.д.), `NN` = порядковый номер.
* **Название**
    * Краткое, но емкое наименование функции, описывающее ее основную суть.
* **Описание**
    * Полное определение требования. Разъясняет, что система должна делать, кто является пользователем, и какие основные шаги или правила вовлечены.
* **Приоритет**
    * Определяет важность требования для проекта и его критичность для основной функциональности. Этот атрибут используется для определения порядка реализации.
    * **Высокий:** Критически важное требование. Без его реализации система не сможет выполнять свои основные бизнес-задачи (например, создание заявки, бронирование). Должно быть реализовано в первой версии.
    * **Средний:** Важное требование. Влияет на ключевые, но не критические функции. Система может работать с ограничениями, но основной бизнес-процесс будет нарушен или неудобен (например, просмотр оценок, уведомления).
    * **Низкий:** Желательное требование. Касается второстепенных функций, удобства использования или расширенной отчетности. Его отсутствие не блокирует основной функционал.

## Надежность
Требования к надежности определяют способность системы выполнять свои функции без сбоев.
- **Доступность**: Система должна быть доступна **99.5%** времени в течение стандартных рабочих часов (9:00 - 18:00 по будням). Допускаются плановые окна обслуживания в нерабочее время (например, суббота с 22:00 до 00:00).
- **MTBF**: Среднее время непрерывной работы системы между двумя критическими сбоями должно составлять не менее 10 000 часов.
- **MTTR**: 1 час. Критические сбои, делающие систему полностью недоступной, должны быть устранены в течение 1 часа в рабочее время.
- **Точность**: Система должна обеспечивать 100% точность хранения и отображения данных о бронированиях, отзывах и оценках.
- **Макс. количество ошибок**: Не более 5 "Критических" или "Высоких" дефектов, блокирующих основной функционал, на момент релиза.

## Производительность
Требования к производительности определяют, насколько быстро система реагирует на действия пользователя при определенной нагрузке.
- **Время отклика**: 
    * Загрузка страниц и отображение списков (заявки, сессии): не более **2 секунд**.
    * Выполнение транзакций (создание заявки, бронирование, отправка отзыва): не более **500 мс**.
- **Пропускная способность**: Система должна поддерживать одновременную работу до **150** активных пользователей без деградации производительности.
- **Утилизация ресурсов**: При пиковой нагрузке (150 одновременных пользователей) утилизация CPU и RAM сервера приложения не должна превышать **80%**.

## Ремонтопригодность
Требования к ремонтопригодности определяют легкость, с которой система может быть модифицирована или исправлена.

* Система должна быть разработана по модульному принципу, чтобы изменения в одном модуле (например, "Отзывы") минимально затрагивали другие (например, "Бронирование").
* Код должен соответствовать внутренним стандартам кодирования компании и быть документирован (inline-комментарии для сложной логики).
* Процесс развертывания (деплоя) исправлений (hotfix) на рабочем сервере не должен занимать более **30 минут** и не должен требовать полной остановки системы.

## Ограничения проекта
Ограничения проекта — это факторы, которые ограничивают выбор решений при разработке.

* **Технологический стек:** Продукт должен быть разработан с использованием текущего технологического стека компании (например, Java/Spring или C#/.NET Core).
* **Аутентификация (SSO):** Аутентификация пользователей должна осуществляться через **корпоративную учетную запись**. Система обязательно должна интегрироваться с корпоративной службой каталогов (например, Active Directory/LDAP).
* **Среда:** Система должна быть веб-приложением, доступным через современные браузеры (Google Chrome, Firefox, MS Edge).
* **Бюджет и сроки:** Проект должен быть завершен в течение 6 месяцев с командой из 4 разработчиков.
* **Данные:** Система не должна хранить персональные данные, выходящие за рамки необходимого (ФИО, e-mail, должность, команда), и должна соответствовать внутренним политикам безопасности данных компании.

## Интерфейсы
- **Интерфейс пользователя**: 
**Тип**: Адаптивный веб-интерфейс (Responsive Web Design), обеспечивающий корректное отображение и работу как на настольных компьютерах, так и на планшетах.
**Стандарты**: Интерфейс должен быть реализован с использованием HTML5, CSS3 и JavaScript (ES6+).
**Браузеры**: Система должна корректно работать в последних стабильных версиях браузеров: Google Chrome, Mozilla Firefox, Microsoft Edge.
**Доступность**: Интерфейс должен быть разработан с учетом базовых принципов доступности (WCAG 2.1 Level AA), таких как корректная семантика HTML, достаточная контрастность текста, возможность навигации с клавиатуры.
- **Аппаратные интерфейсы**: 
 Не применимо. Система является чисто программным веб-решением и не предполагает прямого взаимодействия со специализированным аппаратным обеспечением. Все требования к аппаратному обеспечению относятся к серверной инфраструктуре, на которой система развертывается.
- **Программные интерфейсы**:
 Интеграция с Active Directory / LDAP: Система должна аутентифицировать пользователей через корпоративную службу каталогов по протоколу LDAP/LDAPS. Интерфейс обмена: протокол LDAP версии 3.
 REST API: Система должна предоставлять RESTful API для потенциальной будущей интеграции с другими корпоративными системами (например, с порталом компании, системой календарей). API должен использовать формат данных JSON и аутентификацию на основе токенов (JWT). Детальная спецификация API будет разработана на следующем этапе.
- **Интерфейсы коммуникаций**: 
**Протокол**: Все взаимодействие между клиентом (браузером) и сервером должно осуществляться по защищенному протоколу HTTPS (TLS 1.2 или выше).
**Порты**: Стандартные порты для HTTPS (443).
**Кодировка**: Повсеместно используется кодировка UTF-8 для поддержки кириллицы и других символов.
