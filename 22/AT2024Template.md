# 1. Введение

## Цели

Документ формализует требования к системе мониторинга состояния сотрудников на опасном производстве (Дзержинская ТЭЦ) и соответствует структуре и принципам IEEE Std 830‑1993.

## Границы применения

Система предназначена для мониторинга ЭКГ, движения сотрудников и сигналов тревожной кнопки в реальном времени с отображением и оповещением через панель начальника смены.

## Термины и аббревиатуры

- **ЭКГ** — электрокардиограмма
    
- **ТЭЦ** — теплоэлектроцентраль
    
- **Акселерометр** — датчик движения
    
- **API** — интерфейс программирования приложений
    
- **MQTT** — протокол публикации/подписки для IoT
    
- **HTTP** — протокол передачи гипертекста
    
- **UI** — пользовательский интерфейс
    
- **JSON** — текстовый формат обмена данными
    
- **Wi‑Fi** — беспроводная сеть передачи данных
    
- **BLE** — Bluetooth Low Energy
    
- **REST** — архитектурный стиль взаимодействия компонентов по HTTP
    
- **IoT** — интернет вещей
    
- **ESP32** — микроконтроллер с Wi‑Fi и Bluetooth
    
- **CPU** — центральный процессор
    
- **RAM** — оперативная память
    
- **ORM** — объектно‑реляционное отображение

# 2. Общее описание

## Описание изделия

Система состоит из носимых устройств, серверного компонента и панели оператора и предназначена для обнаружения и оперативного реагирования на критические состояния здоровья сотрудников.

- **Интерфейсы системы**: серверный компонент, панель оператора и носимые устройства, связанные через Wi‑Fi/BLE и протоколы MQTT/WebSocket/HTTP REST.
    
- **Интерфейсы пользователя**: веб‑панель начальника смены для мониторинга статусов сотрудников и оповещения.
    
- **Интерфейсы аппаратных средств ЭВМ**: сервер на базе x86‑совместимого процессора с ОС Ubuntu и поддержкой Docker.
    
- **Интерфейсы программного обеспечения**: REST API для интеграции с СКД и другими системами безопасности предприятия.
    
- **Интерфейсы коммуникаций**: внутренняя Wi‑Fi‑сеть для производственных помещений и резервный BLE‑канал между устройствами и шлюзами.
    

## Функции изделия

- Сбор и передача данных ЭКГ и акселерометра от каждого сотрудника
    
- Обработка данных в реальном времени
    
- Реакция на тревожную кнопку
    
- Звуковая и визуальная индикация на месте происшествия и на панели мониторинга
    
- Интеграция с системой контроля допуска для активации мониторинга
    

## Характеристики пользователей

- **Начальник смены** — оператор панели мониторинга.  
- **Сотрудники смены** — носят устройство мониторинга.  
- **Служба реагирования** — получает уведомления.  
- **Администратор СКД** — управляет привязкой устройств к сотрудникам.

## Ограничения

- Непрерывная работа 24/7, доступность системы ≥ 99.99%.  
- Максимальное время восстановления критических компонентов ≤ 10 минут.  
- Суммарная задержка от события до отображения на панели ≤ 10 секунд.

## Предположения и зависимости

- На сотрудников можно безопасно крепить датчики.  
- Доступ к корпоративной сети передачи данных гарантирован.  
- СКД предоставляет документированный API для интеграции.  
- Соблюдаются правила эксплуатации оборудования на опасных участках.

# 3. Детальные требования

## Функциональные требования

### Функции носимого устройства (Wearable)

**W-F-1. Мониторинг ЭКГ**  
Носимое устройство должно непрерывно измерять ЭКГ сотрудника и передавать данные на сервер с заданным интервалом (например, 1–5 секунд).

- Компонент: носимое устройство
    
- Критичность: высокая
    
- Стабильность: высокая
    
- Тип: функциональное, системное
    

**W-F-2. Детекция движения и падений**  
Носимое устройство должно измерять ускорения по осям и выявлять события падения или отсутствия движения по заданным пороговым правилам.

- Компонент: носимое устройство
    
- Критичность: высокая
    
- Стабильность: средняя
    
- Тип: функциональное, алгоритмическое
    

**W-F-3. Тревожная кнопка**  
Носимое устройство должно иметь физическую тревожную кнопку, нажатие которой фиксируется и в течение не более 2 секунд генерирует событие тревоги, отправляемое на сервер.

- Компонент: носимое устройство
    
- Критичность: высокая
    
- Стабильность: высокая
    
- Тип: функциональное, интерфейс пользователя
    

**W-F-4. Звуковая индикация на устройстве**  
Носимое устройство должно обеспечивать звуковую индикацию нештатных ситуаций тремя различимыми сигналами:

- прерывистый — срабатывание по пороговым значениям (ЭКГ/движение);
    
- непрерывный — активация тревожной кнопки;
    
- периодический — отказ устройства или канала связи.
    
- Компонент: носимое устройство
    
- Критичность: средняя
    
- Стабильность: высокая
    
- Тип: функциональное, интерфейс пользователя
    

**W-F-5. Heartbeat‑сообщения**  
Носимое устройство должно периодически передавать на сервер своё состояние (норма/пограничное/критическое/отказ) в виде heartbeat‑сообщений.

- Компонент: носимое устройство
    
- Критичность: высокая
    
- Стабильность: средняя
    
- Тип: функциональное, диагностическое
    

**W-F-6. Резервный канал связи**  
Носимое устройство должно уметь переключаться на резервный канал связи (BLE) при недоступности основной Wi‑Fi‑сети.

- Компонент: носимое устройство
    
- Критичность: средняя
    
- Стабильность: низкая
    
- Тип: функциональное, коммуникационное
    

---

### Функции сервера (Server)

**SRV-F-1. Приём и хранение данных**  
Сервер должен принимать потоки данных ЭКГ и движения от всех активных носимых устройств в реальном времени и сохранять их в базе данных для последующего анализа.

- Компонент: сервер
    
- Критичность: высокая
    
- Стабильность: высокая
    
- Тип: функциональное, хранилище данных
    

**SRV-F-2. Анализ данных и генерация событий**  
Сервер должен в режиме реального времени анализировать входящие данные (ЭКГ, движение, статусы устройств) и формировать события: «пограничное состояние», «критическое состояние», «тревожная кнопка», «отказ устройства».

- Компонент: сервер
    
- Критичность: высокая
    
- Стабильность: средняя
    
- Тип: функциональное, бизнес‑логика
    

**SRV-F-3. Оповещение панели оператора**  
Сервер должен обеспечивать передачу уведомлений о событиях на панель оператора с суммарной задержкой системы не более 10 секунд с момента возникновения события на носимом устройстве.

- Компонент: сервер
    
- Критичность: высокая
    
- Стабильность: высокая
    
- Тип: функциональное, производительность
    

**SRV-F-4. Интеграция с СКД**  
Сервер должен интегрироваться с системой контроля допуска (СКД) по согласованному API:

- получать информацию о сотруднике и его нахождении на смене;
    
- активировать/деактивировать мониторинг соответствующего носимого устройства.
    
- Компонент: сервер
    
- Критичность: высокая
    
- Стабильность: средняя
    
- Тип: функциональное, интеграционное
    

**SRV-F-5. Внешние API для мониторинга**  
Сервер должен предоставлять программный интерфейс (REST API/MQTT/WebSocket) для:

- получения текущего состояния сотрудников и устройств;
    
- просмотра истории событий и данных мониторинга.
    
- Компонент: сервер
    
- Критичность: средняя
    
- Стабильность: высокая
    
- Тип: функциональное, внешние интерфейсы
    

**SRV-F-6. Обнаружение отказа устройств**  
Сервер должен автоматически определять отсутствие heartbeat‑сообщений от носимого устройства в течение заданного таймаута и генерировать событие «отказ устройства».

- Компонент: сервер
    
- Критичность: высокая
    
- Стабильность: средняя
    
- Тип: функциональное, диагностическое
    

---

### Функции панели оператора (Operator Panel)

**OP-F-1. Отображение статусов сотрудников**  
Панель оператора должна отображать всех сотрудников смены с цветовой индикацией статуса:

- зелёный — норма;
    
- жёлтый — пограничные показатели;
    
- красный — критические показатели/тревожная кнопка;
    
- чёрный — отказ устройства.
    
- Компонент: панель оператора
    
- Критичность: высокая
    
- Стабильность: высокая
    
- Тип: функциональное, UI
    

**OP-F-2. Обновление в реальном времени**  
Панель оператора должна обновлять статусы сотрудников и новые события без ручного обновления страницы (push‑механизм или WebSocket).

- Компонент: панель оператора
    
- Критичность: высокая
    
- Стабильность: высокая
    
- Тип: функциональное, UI/реального времени
    

**OP-F-3. Звуковая индикация на панели**  
Панель оператора должна сопровождать события звуковой индикацией, различая типы событий (пороговые показатели, тревожная кнопка, отказ устройства) и обеспечивая настройку громкости в соответствии с политиками предприятия.

- Компонент: панель оператора
    
- Критичность: средняя
    
- Стабильность: средняя
    
- Тип: функциональное, UI
    

**OP-F-4. Журнал событий**  
Панель оператора должна предоставлять оператору доступ к журналу событий с фильтрацией по сотруднику, типу события и интервалу времени.

- Компонент: панель оператора
    
- Критичность: средняя
    
- Стабильность: высокая
    
- Тип: функциональное, отчётность
    

**OP-F-5. Диагностика устройств**  
Панель оператора должна отображать для каждого носимого устройства индикатор технического состояния (уровень сигнала, время последней связи, уровень заряда — при наличии данных).

- Компонент: панель оператора
    
- Критичность: средняя
    
- Стабильность: средняя
    
- Тип: функциональное, диагностическое

## Нефункциональные требования

### Надёжность (Reliability)

**REL-NF-1. Доступность системы**  
Система (сервер + панель оператора) должна обеспечивать доступность не менее 99.99% при круглосуточной эксплуатации 24/7.

- Компонент: система
    
- Критичность: высокая
    
- Стабильность: высокая
    
- Тип: нефункциональное, надёжность
    

**REL-NF-2. Восстановление после отказа**  
Максимальное время восстановления после отказа критических компонент (сервер, БД, сетевые сервисы) не должно превышать 10 минут.

- Компонент: система
    
- Критичность: высокая
    
- Стабильность: средняя
    
- Тип: нефункциональное, ремонтопригодность
    

**REL-NF-3. Резервирование сервисов**  
Система должна обеспечивать автоматическое резервирование или перезапуск критических сервисов сервера (контейнеры, службы) для минимизации простоя.

- Компонент: сервер
    
- Критичность: средняя
    
- Стабильность: средняя
    
- Тип: нефункциональное, отказоустойчивость
    

**REL-NF-4. Надёжность носимого устройства**  
Носимое устройство должно обеспечивать устойчивую работу без перезагрузки в течение одной смены (длительность смены подлежит уточнению) при соблюдении условий эксплуатации.

- Компонент: носимое устройство
    
- Критичность: высокая
    
- Стабильность: низкая
    
- Тип: нефункциональное, надёжность

### Производительность (Performance)

**PERF-NF-1. Задержка отправки события**  
Время задержки между регистрацией критического события на носимом устройстве (порог ЭКГ, падение, тревожная кнопка) и отправкой сообщения на сервер не должно превышать 2 секунд.

- Компонент: носимое устройство
    
- Критичность: высокая
    
- Стабильность: средняя
    
- Тип: нефункциональное, производительность
    

**PERF-NF-2. Обработка сервером событий**  
Время обработки критического события на сервере и формирования сообщения для панели оператора не должно превышать 3 секунд.

- Компонент: сервер
    
- Критичность: высокая
    
- Стабильность: средняя
    
- Тип: нефункциональное, производительность
    

**PERF-NF-3. Масштабируемость**  
Сервер должен поддерживать одновременную обработку данных от заданного количества носимых устройств (значение N подлежит уточнению) без превышения суммарного времени реакции системы более 10 секунд от события до отображения на панели.

- Компонент: сервер
    
- Критичность: высокая
    
- Стабильность: низкая
    
- Тип: нефункциональное, масштабируемость
    

**PERF-NF-4. Обновление UI панели**  
Панель оператора должна обновлять отображение статуса сотрудников после получения события от сервера не более чем за 1 секунду.

- Компонент: панель оператора
    
- Критичность: средняя
    
- Стабильность: высокая
    
- Тип: нефункциональное, производительность UI
    

---

### Ремонтопригодность (Maintainability)

**MAINT-NF-1. Централизованное логирование**  
Система должна поддерживать централизованное логирование на сервере (события приложений, ошибки, интеграционные ошибки) с возможностью фильтрации по устройству, сотруднику и типу события.

- Компонент: сервер
    
- Критичность: средняя
    
- Стабильность: высокая
    
- Тип: нефункциональное, эксплуатационное
    

**MAINT-NF-2. Обновление прошивки устройств (OTA)**  
Носимые устройства должны поддерживать удалённое обновление прошивки (OTA), инициируемое с сервера при соблюдении требований информационной безопасности предприятия.

- Компонент: носимое устройство + сервер
    
- Критичность: средняя
    
- Стабильность: средняя
    
- Тип: нефункциональное, ремонтопригодность
    

**MAINT-NF-3. Диагностические сообщения на панели**  
Панель оператора должна отображать диагностические сообщения (ошибка связи с устройством, ошибка интеграции с СКД, ошибка БД) для упрощения анализа инцидентов службой поддержки.

- Компонент: панель оператора
    
- Критичность: средняя
    
- Стабильность: высокая
    
- Тип: нефункциональное, эксплуатационное

- Компонент: носимое устройство
    
- Критичность: средняя

- Тип: ограничение реализации

- Компонент: сервер
    
- Критичность: средняя
    
- Стабильность: высокая
    
- Тип: ограничение инфраструктуры
    

**CONST-NF-3. Доступ к панели оператора**  
Панель оператора должна быть доступна через стандартный веб‑браузер в локальной сети предприятия без установки дополнительного ПО.

- Компонент: панель оператора
    
- Критичность: средняя
    
- Стабильность: высокая
    
- Тип: ограничение среды

## Интерфейсы

- **Интерфейс пользователя**: Панель мониторинга отображает статус сотрудников с цветовой кодировкой:
  - Зеленый — норма
  - Желтый — пограничное состояние
  - Красный — критическое состояние/тревожная кнопка
  - Черный — отказ устройства
  Звуковые сигналы различаются для различных типов событий
- **Аппаратные интерфейсы**: Устройства сотрудников: ЭКГ, акселерометр, тревожная кнопка, звуковые индикаторы
- **Программные интерфейсы**: Интеграция с системой безопасности предприятия и системой контроля допуска
- **Интерфейсы коммуникаций**: 
  - Передача данных в реальном времени по протоколам MQTT, WebSocket, HTTP REST API
  - Развертка выделенной Wi-Fi сети для обеспечения стабильного покрытия всех производственных помещений
  - Резервный канал связи через BLE
