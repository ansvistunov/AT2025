## 1. Введение

### Введение
Настоящий документ описывает функциональные и нефункциональные требования к программному обеспечению системы контроля и управления доступом (СКУД). Документ предназначен для команды разработки, тестировщиков и руководства проекта.

### Цели
Цель документа — обеспечить однозначное понимание функционала, ограничений и характеристик разрабатываемой системы всеми участниками проекта. Документ служит основой для проектирования, разработки и приемочного тестирования.

### Границы применения
Документ распространяется на серверную часть системы, клиентское ПО для рабочих мест операторов/администраторов, а также микропрограммное обеспечение (прошивки) для контроллеров, управляющих периферийными устройствами (считывателями, замками).

### Термины и аббревиатуры
| Термин/Аббревиатура | Определение |
| :--- | :--- |
| СКУД | Система Контроля и Управления Доступом |
| Уровень доступа | Категория помещений, определяющая базовый набор зон, которые может посещать сотрудник (напр., "Бухгалтерия", "Склад", "IT"). |
| Маска доступа | Дополнительный набор разрешений, накладываемый поверх уровня доступа для предоставления прав на посещение конкретных помещений вне списка уровня. |
| Ключ-карта | Уникальный идентификатор (карта, брелок), принадлежащий сотруднику. |
| Контактный считыватель | Устройство, требующее явного прикладывания ключ-карты. |
| Дистанционный считыватель | Устройство, автоматически обнаруживающее и идентифицирующее ключ-карты в своей зоне действия. |

| Периметр | Внешние границы охраняемого объекта (двери, ворота). |

## 2. Общее описание

### Описание изделия
Система представляет собой распределенную программно-аппаратную платформу для управления доступом персонала на территорию предприятия. Система в реальном времени отслеживает перемещения сотрудников, проверяет права доступа, управляет исполнительными устройствами и оповещает службу безопасности о нарушениях.

*   **Интерфейсы пользователя:** Веб-интерфейс для администратора и оператора охраны. Графическая карта объекта с отображением статуса дверей, тревог и местоположения сотрудников.
*   **Интерфейсы аппаратных средств ЭВМ:** Система взаимодействует с аппаратными контроллерами через промышленные сети (RS-485, Ethernet, CAN).
*   **Интерфейсы программного обеспечения:** REST API для интеграции со смежными системами (например, система учета рабочего времени).
*   **Интерфейсы коммуникаций:** Поддержка TCP/IP для внутренней связи между сервером и клиентами.
*   **Ограничения памяти:** Серверная база данных должна быть рассчитана на хранение истории событий не менее чем за 1 год для объекта с численностью персонала до 5000 человек.
*   **Действия:** Система должна обеспечивать круглосуточную непрерывную работу.

### Функции изделия
1.  **Управление доступом:** Проверка прав сотрудника при попытке прохода через контролируемую точку.
2.  **Учет и мониторинг:** Ведение журнала всех событий (проходы, тревоги, действия оператора) и отображение текущего статуса системы.
3.  **Централизованное управление:** Настройка уровней доступа, масок, расписаний и прав пользователей.
4.  **Мониторинг присутствия:** Отслеживание текущего местоположения сотрудников на территории с использованием дистанционных считывателей.
5.  **Сигнализация:** Мгновенное оповещение оператора о нарушениях (проникновение в запрещенную зону, "хвостинг", срабатывание датчика периметра).
6.  **Аварийное управление:** Централизованный перевод всех точек прохода в аварийный режим "Выход открыт".

### Характеристики пользователей
| Класс пользователей | Описание | Ключевые потребности |
| :--- | :--- | :--- |
| **Администратор системы** | Технический специалист | Настройка системы, управление пользователями, правами доступа, генерация отчетов. |
| **Оператор охраны** | Сотрудник службы безопасности | Мониторинг текущей ситуации, реагирование на тревоги, ручное управление дверьми. |
| **Сотрудник** | Конечный пользователь | Проход через контролируемые точки доступа с использованием ключ-карты. |

### Ограничения
*   Система должна работать с существующим парком аппаратных считывателей и контроллеров, поддерживающим стандартные протоколы (например, Wiegand).
*   Время восстановления системы после сбоя не должно превышать 5 минут.
*   Интерфейс оператора должен быть оптимизирован для работы на мониторах с разрешением не менее 1920x1080.

### Предположения и зависимости
*   Предполагается, что аппаратное обеспечение (контроллеры, считыватели, датчики) исправно и имеет стабильное соединение с сервером.
*   Работа системы зависит от наличия стабильного электропитания и сетевой инфраструктуры.

## 3. Детальные требования

### Функциональные требования

**FR-1: Управление доступом на точке прохода**
*   **FR-1.1:** При прикладывании ключ-карты к контактному считывателю система должна проверить уровень доступа и маску сотрудника.
*   **FR-1.2:** Если доступ разрешен, система должна подать команду на исполнительное устройство (разблокировать замок) на заданное время (например, 5 секунд).
*   **FR-1.3:** Если доступ запрещен, система должна инициировать звуковую и/или световую сигнализацию на точке прохода и зарегистрировать событие "Отказ в доступе".

**FR-2: Мониторинг присутствия**
*   **FR-2.1:** Система должна в реальном времени отслеживать и отображать на карте объекта зону нахождения сотрудника на основе сигналов от дистанционных считывателей.
*   **FR-2.2:** Если сотрудник обнаружен в зоне, не разрешенной его уровнем доступа и маской, система должна зарегистрировать событие "Нарушение зоны" и передать тревогу на пульт оператора.

**FR-3: Обнаружение "хвостинга"**
*   **FR-3.1:** При срабатывании датчика движения внутри помещения после единичного события успешного прохода система должна зарегистрировать событие "Предполагаемый хвостинг" и передать тревогу на пульт оператора.

**FR-4: Управление аварийными режимами**
*   **FR-4.1:** При активации аварийного режима (вручную оператором или по сигналу от системы пожарной сигнализации) система должна разблокировать все исполнительные устройства в режиме "свободный выход".
*   **FR-4.2:** В аварийном режиме все проверки доступа на вход должны быть приостановлены.

**FR-5: Реагирование на датчики периметра**
*   **FR-5.1:** При срабатывании датчика контроля периметра (например, открытие двери) система должна немедленно передать тревогу на пульт оператора, указав зону срабатывания.
