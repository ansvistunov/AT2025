## 1. Введение

### Введение
Настоящий документ описывает функциональные и нефункциональные требования к программному обеспечению системы контроля и управления доступом (СКУД). Документ предназначен для команды разработки, тестировщиков и руководства проекта.

### Цели
Цель документа — обеспечить однозначное понимание функционала, ограничений и характеристик разрабатываемой системы всеми участниками проекта. Документ служит основой для проектирования, разработки и приемочного тестирования.

### Границы применения
Документ распространяется на серверную часть системы, клиентское ПО для рабочих мест операторов/администраторов, а также микропрограммное обеспечение (прошивки) для контроллеров, управляющих периферийными устройствами (считывателями, замками, сиренами).

### Термины и аббревиатуры
| Термин/Аббревиатура | Определение |
| :--- | :--- |
| СКУД | Система Контроля и Управления Доступом |
| Уровень доступа | Категория помещений, определяющая базовый набор зон, которые может посещать сотрудник (напр., "Бухгалтерия", "Склад", "IT"). |
| Маска доступа | Дополнительный набор разрешений, накладываемый поверх уровня доступа для предоставления прав на посещение конкретных помещений вне списка уровня. |
| Ключ-карта | Уникальный идентификатор (карта, брелок) с радиометкой, принадлежащий сотруднику. |
| Контактный считыватель | Устройство, требующее явного прикладывания ключ-карты. |
| Дистанционный считыватель | Устройство, автоматически обнаруживающее и идентифицирующее ключ-карты в своей зоне действия. |

| Периметр | Внешние границы охраняемого объекта (двери, ворота). |

## 2. Общее описание

### Описание изделия
Система представляет собой распределенную программно-аппаратную платформу для управления доступом персонала на территорию предприятия. Система в реальном времени отслеживает перемещения сотрудников с помощью радиометок, проверяет права доступа, управляет исполнительными устройствами и оповещает службу безопасности о нарушениях. Интегрирована с системой пожарной сигнализации и автоматического пожаротушения.

*   **Интерфейсы пользователя:** Веб-интерфейс для администратора и оператора охраны. Графическая карта объекта с отображением статуса дверей, тревог и местоположения сотрудников.
*   **Интерфейсы аппаратных средств ЭВМ:** Система взаимодействует с аппаратными контроллерами через промышленные сети (RS-485, Ethernet, CAN). Интерфейс с радиометками (RFID 125 кГц/13.56 МГц).
*   **Интерфейсы программного обеспечения:** REST API для интеграции со смежными системами (пожарная сигнализация, система учета рабочего времени).
*   **Интерфейсы коммуникаций:** Поддержка TCP/IP для внутренней связи между сервером и клиентами.
*   **Ограничения памяти:** Серверная база данных должна быть рассчитана на хранение истории событий не менее чем за 1 год для объекта с численностью персонала до 5000 человек.
*   **Действия:** Система должна обеспечивать круглосуточную непрерывную работу.

### Функции изделия
1.  **Управление доступом:** Проверка прав сотрудника при попытке прохода через контролируемую точку.
2.  **Трекинг местоположения:** Автоматическое отслеживание нахождения сотрудников с помощью радиометок и сетки считывающих устройств.
3.  **Учет и мониторинг:** Ведение журнала всех событий (проходы, тревоги, действия оператора) и отображение текущего статуса системы.
4.  **Централизованное управление:** Настройка уровней доступа, масок, расписаний и прав пользователей.
5.  **Реагирование на нарушения:** Активация исполнительных устройств (сирены) при несанкционированном доступе.
6.  **Интеграция с пожарной системой:** Автоматическое переключение режимов работы при пожаре, управление автоматическим пожаротушением.
7.  **Управление режимами работы:** Централизованное переключение между основным и аварийным режимами.

### Характеристики пользователей
| Класс пользователей | Описание | Ключевые потребности |
| :--- | :--- | :--- |
| **Администратор системы** | Технический специалист | Настройка системы, управление пользователями, правами доступа, генерация отчетов. |
| **Оператор охраны** | Сотрудник службы безопасности | Мониторинг текущей ситуации, реагирование на тревоги, ручное управление дверьми и режимами. |
| **Сотрудник** | Конечный пользователь | Проход через контролируемые точки доступа с использованием ключ-карты. |

### Ограничения
*   Система должна работать с существующим парком аппаратных считывателей и контроллеров, поддерживающим стандартные протоколы (например, Wiegand, RFID).
*   Время восстановления системы после сбоя не должно превышать 5 минут.
*   Интерфейс оператора должен быть оптимизирован для работы на мониторах с разрешением не менее 1920x1080.

### Предположения и зависимости
*   Предполагается, что аппаратное обеспечение (контроллеры, считыватели, датчики, радиометки) исправно и имеет стабильное соединение с сервером.
*   Работа системы зависит от наличия стабильного электропитания и сетевой инфраструктуры.
*   Предполагается наличие интерфейса для интеграции с системой пожарной сигнализации и автоматического пожаротушения.

## 3. Детальные требования

### 3.1 Функциональные требования

#### FR-1: Управление доступом на точке прохода
**Описание:** Система должна проверять права доступа сотрудника при попытке прохода через контролируемую точку и управлять исполнительными устройствами.

**Детали:**
- **FR-1.1:** При прикладывании ключ-карты к контактному считывателю система должна проверить уровень доступа и маску сотрудника.
- **FR-1.2:** Если доступ разрешен, система должна подать команду на разблокирование замка на заданное время (5-10 секунд).
- **FR-1.3:** Если доступ запрещен, система должна инициировать звуковую/световую сигнализацию на точке прохода.

**Атрибуты:**
| Атрибут | Значение | Описание |
| :--- | :--- | :--- |
| **Приоритет** | Высокий | Критически важное требование |
| **Сложность** | Средняя | Требует интеграции с аппаратным обеспечением |
| **Статус** | Запланировано | |
| **Источник** | Описание системы | Основано на основном режиме работы |

---

#### FR-2: Мониторинг присутствия и обнаружение нарушителей
**Описание:** Система должна отслеживать местоположение сотрудников с помощью дистанционных считывателей и обнаруживать нахождение в запрещенных зонах.

**Детали:**
- **FR-2.1:** Система должна в реальном времени определять зону нахождения сотрудника по сигналам дистанционных считывателей.
- **FR-2.2:** При обнаружении сотрудника в зоне, не разрешенной его уровнем доступа и маской, система должна генерировать тревогу.

**Атрибуты:**
| Атрибут | Значение | Описание |
| :--- | :--- | :--- |
| **Приоритет** | Высокий | Ключевая функция безопасности |
| **Сложность** | Высокая | Требует точного позиционирования и логики анализа |
| **Статус** | Запланировано | |
| **Бизнес-ценность** | Критическая | Прямо влияет на безопасность объекта |

---

#### FR-3: Обнаружение "хвостинга" (Tailgating)
**Описание:** Система должна обнаруживать случаи, когда один человек проходит по чужой ключ-карте.

**Детали:**
- **FR-3.1:** При срабатывании датчика движения внутри помещения после единичного события успешного прохода система должна регистрировать "Предполагаемый хвостинг".
- **FR-3.2:** Тревога "хвостинг" должна передаваться на пульт оператора с указанием точки доступа.

**Атрибуты:**
| Атрибут | Значение | Описание |
| :--- | :--- | :--- |
| **Приоритет** | Средний | Важная, но не критическая функция |
| **Сложность** | Средняя | Требует анализа последовательности событий |
| **Статус** | Запланировано | |
| **Валидация** | Ручное тестирование | Требует имитации сценария |

---

#### FR-4: Управление аварийными режимами
**Описание:** Система должна обеспечивать экстренную разблокировку всех точек доступа при чрезвычайных ситуациях.

**Детали:**
- **FR-4.1:** При активации аварийного режима система должна разблокировать все исполнительные устройства в режиме "свободный выход".
- **FR-4.2:** Активация может выполняться вручную оператором или автоматически по сигналу от системы пожарной сигнализации.

**Атрибуты:**
| Атрибут | Значение | Описание |
| :--- | :--- | :--- |
| **Приоритет** | Критический | Влияет на жизнь и здоровье людей |
| **Сложность** | Низкая | Централизованное управление состоянием |
| **Статус** | Запланировано | |
| **Связанные требования** | Безопасность, надежность | |

#### FR-5: Реагирование на датчики периметра
**Описание:** Система должна немедленно реагировать на срабатывание датчиков контроля периметра.

**Детали:**
- **FR-5.1:** При срабатывании датчика периметра (открытие двери, разбитие стекла) система должна немедленно передавать тревогу на пульт оператора.
- **FR-5.2:** В интерфейсе оператора должна подсвечиваться зона срабатывания на карте объекта.

**Атрибуты:**
| Атрибут | Значение | Описание |
| :--- | :--- | :--- |
| **Приоритет** | Высокий | Защита периметра - первоочередная задача |
| **Сложность** | Низкая | Обработка дискретных сигналов |
| **Статус** | Запланировано | |
| **Производительность** | Реальное время | Задержка не более 1 секунды |

#### FR-6: Управление учетными записями и правами доступа
**Описание:** Система должна предоставлять инструменты для управления сотрудниками, уровнями доступа и масками.

**Детали:**
- **FR-6.1:** Администратор должен иметь возможность создавать, редактировать и удалять учетные записи сотрудников.
- **FR-6.2:** Должна быть возможность назначать уровни доступа и дополнительные маски для каждого сотрудника.
- **FR-6.3:** Система должна поддерживать неограниченное количество уровней доступа.

**Атрибуты:**
| Атрибут | Значение | Описание |
| :--- | :--- | :--- |
| **Приоритет** | Высокий | Базовый функционал администрирования |
| **Сложность** | Средняя | Сложная логика наследования прав |
| **Статус** | Запланировано | |
| **Пользователи** | Администратор | |

#### FR-7: Ведение журнала событий и отчетность
**Описание:** Система должна регистрировать все события и предоставлять инструменты для анализа.

**Детали:**
- **FR-7.1:** Все события (проходы, тревоги, действия оператора) должны записываться в базу данных.
- **FR-7.2:** Должна быть возможность фильтрации и поиска по журналу событий.
- **FR-7.3:** Система должна предоставлять стандартные отчеты (отчет по проходам, отчет по тревогам).

**Атрибуты:**
