# 1. Введение

Настоящий документ содержит спецификацию требований к программному обеспечению для системы дистанционного обучения с механизмом взаимной проверки работ. Документ описывает функциональные и нефункциональные требования, ограничения и интерфейсы системы.

## Цели

Основной целью системы является решение проблемы массовой оценки студенческих работ в условиях дистанционного обучения, когда один преподаватель обучает сотни или тысячи студентов. Система автоматизирует процесс взаимной проверки работ студентами с контролем качества оценок через анализ дисперсии.

## Границы применения

Система предназначена для образовательных учреждений, реализующих программы дистанционного обучения. В границы системы входит: управление курсами и заданиями, загрузка работ, автоматическое распределение работ для взаимной проверки, расчет дисперсии оценок, формирование итоговых оценок. Система не включает функциональность видеолекций, тестирования или форумов.

## Термины и аббревиатуры

| СДО | Система дистанционного обучения |

| Дисперсия | Статистическая мера разброса оценок |

| Количество проверок | Настраиваемый параметр системы |

# 2. Общее описание

## Описание изделия

- Интерфейсы системы: REST API для интеграции с внешними системами
- Интерфейсы пользователя: Веб-интерфейс для преподавателей и студентов
- Интерфейсы аппаратных средств ЭВМ: Стандартные веб-серверы и системы хранения данных
- Интерфейсы программного обеспечения: Поддержка интеграции с LMS системами
- Интерфейсы коммуникаций: HTTP/HTTPS протоколы, поддержка мобильных устройств
- Ограничения памяти: Поддержка хранения тысяч студенческих работ и оценок
- Действия: Автоматическое распределение работ, расчет дисперсии, уведомления

## Функции изделия

- Создание и управление учебными курсами и заданиями
- Загрузка студенческих работ в различных форматах
- Автоматическое распределение работ для взаимной проверки
- Процесс оценки работ по 5-балльной шкале
- Расчет дисперсии оценок и принятие решений
- Выставление среднего балла или отправка преподавателю
- Управление пользователями и ролями
- Формирование отчетов и аналитика

## Характеристики пользователей

| Профиль | Обязанности |
|---------|-------------|
| Преподаватель | Создание курсов, настройка параметров проверки, проверка спорных работ |
| Студент | Загрузка работ, проверка работ других студентов |
| Администратор | Управление системой и пользователями |

## Ограничения

- Поддержка только веб-интерфейса
- Ограничение на размер загружаемых файлов
- Максимальное количество одновременных пользователей: 10 000
- Совместимость с современными веб-браузерами
- Использование реляционных СУБД

## Предположения и зависимости

- Наличие стабильного интернет-соединения у пользователей
- Добросовестное отношение студентов к проверке работ
- Наличие технической поддержки со стороны организации
- Интеграция с системами аутентификации организации
- Поддержка со стороны IT-инфраструктуры

# 3. Детальные требования

## Функциональные требования


Функциональные требования сгруппированы по ролям пользователей системы для обеспечения ясности и удобства использования.

Каждое функциональное требование описывается с помощью следующих атрибутов:

- ID - уникальный идентификатор требования в формате FR-[Роль]-[Номер], где:
  - FR - Functional Requirement (Функциональное требование)
  - [Роль] - код роли пользователя 
  - [Номер] - порядковый номер требования для данной роли
- Название - краткое описательное название требования, отражающее его суть
- Описание - развернутое описание функциональности, которую должна предоставлять система
- Приоритет - важность реализации требования (Высокий, Средний, Низкий)
- Критерии приемки - конкретные условия, которые должны быть выполнены для подтверждения корректной реализации требования

#### Схема идентификаторов
| Префикс | Расшифровка | Роль |
|---------|-------------|------|
| FR-AD- | Administrator | Администратор системы |
| FR-TC- | Teacher | Преподаватель |
| FR-ST- | Student | Студент |

## Надежность

- Доступность: Система должна обеспечивать доступность 99% в течение учебного семестра
- Безотказность: Среднее время наработки на отказ для основных функций - не менее 168 часов (7 дней)
- Восстанавливаемость: Время восстановления после сбоя - не более 4 часов
- Точность расчетов: Расчет дисперсии и средних оценок должен выполняться с точностью до 0.01 балла
- Целостность данных: Гарантированное сохранение всех загруженных работ и выставленных оценок

## Производительность

- Время отклика:
  - Загрузка списка курсов/заданий - не более 2 секунд
  - Загрузка работы студента - не более 3 секунд
  - Расчет дисперсии оценок - не более 5 секунд
  - Сохранение оценки - не более 1 секунды

- Пропускная способность:
  - Поддержка одновременной работы до 1000 пользователей
  - Обработка до 5000 проверок в день
  - Одновременная загрузка до 100 работ

- Масштабируемость:
  - Возможность увеличения производительности при росте количества пользователей
  - Поддержка кластерной архитектуры для распределения нагрузки

## Ремонтопригодность

- Модульность архитектуры: Возможность обновления отдельных компонентов системы без остановки всей системы
- Логирование: Детальное логирование всех операций для диагностики проблем
- Мониторинг: Встроенные инструменты мониторинга производительности и состояния системы
- Документация: Полная техническая документация по установке, настройке и обслуживанию системы

## Ограничения проекта

- Технологические: Использование только веб-технологий, поддержка современных браузеров
- Безопасность: Обязательная аутентификация пользователей, разграничение прав доступа
- Лицензирование: Использование преимущественно open-source технологий
- Инфраструктурные: Требования к серверному оборудованию и каналам связи
