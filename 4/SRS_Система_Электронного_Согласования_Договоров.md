# 1. Введение
Данный документ содержит спецификацию требований к программному обеспечению (SRS) для системы электронного согласования договоров. Документ определяет функциональные и нефункциональные требования к системе, а также описывает основные характеристики и ограничения разрабатываемого ПО.

## Цели
Целью данного документа является детальное описание требований к системе электронного согласования договоров для обеспечения единого понимания функциональности системы всеми участниками проекта: заказчиками, разработчиками, тестировщиками и пользователями. Документ служит основой для проектирования, разработки и тестирования системы.

## Границы применения
Данный документ распространяется на систему электронного согласования договоров, предназначенную для автоматизации процесса согласования документов внутри организации. Система обеспечивает создание маршрутов согласования, управление версиями документов, уведомление участников процесса и архивирование утверждённых документов.

## Термины и аббревиатуры
| Термин/Аббревиатура           | Определение                                                                     |

# 2. Общее описание

## Описание изделия
- **Интерфейсы системы**: Система представляет собой веб-приложение, доступное через браузер. Взаимодействие с внешними системами предполагается через электронную почту для отправки уведомлений.
- **Интерфейсы пользователя**: Веб-интерфейс с графическим отображением статусов документов, формами загрузки файлов, настройки маршрутов согласования и просмотра истории. Интерфейс должен быть интуитивно понятным и адаптивным для различных устройств.
- **Интерфейсы аппаратных средств ЭВМ**: Доступ к системе осуществляется через стандартные компьютеры, планшеты или смартфоны с современными веб-браузерами.
- **Интерфейсы программного обеспечения**: Система взаимодействует с почтовым сервером для отправки уведомлений по протоколу SMTP. Требуется поддержка просмотра документов различных форматов (PDF, Word, изображения).
- **Интерфейсы коммуникаций**: HTTP/HTTPS для взаимодействия клиента с сервером, SMTP для отправки электронной почты.
- **Ограничения памяти**: Система должна поддерживать загрузку документов размером до 50 МБ. Архив должен обеспечивать хранение документов в течение 10 лет.
- **Действия**: Создание документов, настройка маршрутов согласования, утверждение/отклонение документов, добавление замечаний, поиск и просмотр архивных документов, управление пользователями.

## Функции изделия
Система электронного согласования договоров обеспечивает следующие основные функции:

1. **Управление документами**
   - Загрузка проектов документов различных форматов (PDF, Word, изображения)
   - Хранение версий документов
   - Просмотр документов всеми участниками процесса

2. **Маршрутизация согласования**
   - Создание маршрутов с выбором согласующих лиц
   - Поддержка параллельного согласования (все согласующие работают одновременно)
   - Поддержка последовательного согласования (документ переходит по цепочке)
   - Установка сроков согласования (настраиваемый срок по умолчанию и индивидуальный для каждого документа)

3. **Процесс согласования**
   - Утверждение документов согласующими
   - Отклонение документов с обязательным указанием замечаний
   - Возврат документа инициатору для исправления
   - Повторный запуск согласования после исправлений

4. **Уведомления**
   - Отправка уведомлений по электронной почте о новых документах на согласовании
   - Уведомления об отклонении с указанием замечаний
   - Уведомления об истечении срока согласования

5. **Архивирование и поиск**
   - Автоматическое помещение утверждённых документов в архив
   - Хранение полной истории согласования документа
   - Поиск документов по атрибутам (дата, автор, статус)
   - Полнотекстовый поиск внутри документов

6. **Управление пользователями**
   - Администрирование учётных записей пользователей
   - Назначение ролей (Инициатор, Согласующий, Администратор)
   - Управление правами доступа

7. **Отображение статусов**
   - Визуализация текущего статуса документа (Создан, На согласовании, Утверждён, Отклонён)
   - Отображение прогресса прохождения маршрута согласования

## Характеристики пользователей
Система предназначена для следующих категорий пользователей:

1. **Инициатор**
   - Сотрудники организации, инициирующие процесс согласования договоров
   - Базовые навыки работы с компьютером и веб-приложениями
   - Выполняют функции: создание документов, настройка маршрутов, исправление документов после замечаний

2. **Согласующий**
   - Руководители и ответственные лица, принимающие решения по согласованию
   - Базовые навыки работы с компьютером
   - Выполняют функции: просмотр документов, утверждение или отклонение с указанием замечаний

3. **Администратор**
   - IT-специалисты с техническими навыками
   - Выполняют функции: управление пользователями, назначение ролей, настройка системных параметров

**Примечание**: Один пользователь может одновременно обладать несколькими ролями.

## Ограничения
1. На первом этапе система принимает документы любых форматов, которые могут быть открыты всеми согласующими (PDF, Word, изображения)
2. Ограничений по количеству согласующих в маршруте нет
3. Документы в архиве должны храниться в течение 10 лет
4. Система должна работать в современных веб-браузерах (Chrome, Firefox, Safari, Edge последних версий)
5. Требуется стабильное интернет-соединение для работы с системой
6. Максимальный размер загружаемого файла — 50 МБ

## Предположения и зависимости
1. Предполагается наличие корпоративной почтовой системы для отправки уведомлений
2. Пользователи имеют доступ к современным веб-браузерам и устройствам для просмотра документов
3. Организация предоставит необходимую инфраструктуру для размещения системы (сервер, каналы связи)
4. В будущем планируется добавление функциональности шаблонов документов
5. Система зависит от стабильности сетевой инфраструктуры организации
6. Предполагается интеграция с корпоративной системой управления пользователями (Active Directory или аналог) на последующих этапах развития

# 3. Функциональные требования

## 3.1. Атрибуты функциональных требований

Данный раздел определяет функциональные требования системы электронного согласования документов и описывает поведение системы для каждой категории пользователей: **Инициатор**, **Согласующий**, **Администратор**, а также системные функции, обеспечивающие корректную работу решения.

Каждое функциональное требование должно обладать следующими атрибутами:

- **ID** — уникальный идентификатор требования
- **Наименование** — краткое описание требования
- **Важность** —
  - **M** — обязательное
  - **S** — желательное
  - **C** — возможное
- **Стабильность** —
  - **S** — стабильное
  - **M** — модифицируемое
  - **D** — динамичное

## Надежность
- **Доступность**: 99.5% в рабочее время (с 8:00 до 20:00 по рабочим дням)
- **MTBF (Среднее время между отказами)**: Не менее 720 часов (30 дней)
- **MTTR (Среднее время восстановления)**: Не более 4 часов для критических сбоев
- **Точность**: 100% сохранность данных документов и истории согласования
- **Макс. количество ошибок**: Не более 1 критической ошибки в месяц, не более 5 некритических ошибок в месяц
- **Отказоустойчивость**: Система должна обеспечивать регулярное резервное копирование данных (не реже 1 раза в сутки)
- **Целостность данных**: Все операции с документами должны выполняться транзакционно с возможностью отката в случае сбоя

## Производительность
- **Время отклика**: 
  - Загрузка интерфейса: не более 2 секунд
  - Загрузка документа размером до 10 МБ: не более 5 секунд
  - Поиск документов: не более 3 секунд
  - Выполнение операций согласования/отклонения: не более 1 секунды
- **Пропускная способность**: 
  - Поддержка одновременной работы не менее 100 пользователей
  - Обработка не менее 1000 документов в месяц
- **Утилизация ресурсов**: 
  - Использование оперативной памяти сервера: не более 70% при пиковой нагрузке
  - Использование процессора: не более 80% при пиковой нагрузке
  - Дисковое пространство: планирование на хранение не менее 10000 документов (ориентировочно 500 ГБ)

## Ремонтопригодность
- Система должна вести подробные логи всех операций для диагностики проблем
- Архитектура системы должна быть модульной для упрощения обновления отдельных компонентов
- Код должен быть документирован и соответствовать принятым стандартам разработки
- Система должна предоставлять административный интерфейс для мониторинга состояния и диагностики
- Должна быть предусмотрена возможность горячего резервирования критических компонентов
- Обновления системы должны проводиться без длительных простоев (не более 2 часов в нерабочее время)

## Ограничения проекта
- Разработка должна вестись с использованием современных веб-технологий
- Система должна быть развёрнута на инфраструктуре заказчика (on-premise решение)
- Бюджет проекта и сроки разработки согласовываются отдельно
- На первом этапе не предусмотрена разработка мобильных приложений (только адаптивный веб-интерфейс)
- Интеграция с внешними системами (кроме почтового сервера) не входит в первую версию
- Электронная подпись документов планируется к реализации в следующих версиях

## Требования к пользовательской документации
- **Руководство пользователя**: Подробное описание работы с системой для каждой роли (Инициатор, Согласующий, Администратор)
- **Руководство администратора**: Инструкции по установке, настройке, обслуживанию системы, управлению пользователями
- **Техническая документация**: Архитектура системы, описание API, схемы базы данных
- **Онлайн-справка**: Контекстная справка, доступная из интерфейса системы
- **Учебные материалы**: Краткие инструкции и видео-уроки для быстрого освоения системы
- Вся документация должна быть на русском языке
- Документация должна обновляться при каждом значительном обновлении системы
