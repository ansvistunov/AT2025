# 1. Введение

## Цели
Целью данного документа является определение функциональных и нефункциональных требований к программному обеспечению для системы управления заявками компании, занимающейся аутсорсингом. Документ предназначен для использования командой разработки, тестировщиками и менеджерами проекта в качестве руководства при проектировании, реализации и приемке программного обеспечения. Документ также служит основой для согласования ожиданий с заказчиком.

## Границы применения
Данный документ распространяется на веб-приложение для управления жизненным циклом заявок на обслуживание от клиентов компании-аутсорсера. Система включает в себя портал для клиентов, интерфейс для диспетчера и рабочее место сотрудника (исполнителя). Документ охватывает требования к функционалу приема заявок через несколько каналов (веб-приложение, email, телефон), их маршрутизации, выполнения, уведомления клиентов и обеспечения обратной связи.

# 2. Общее описание

## Описание изделия
Система представляет собой веб-приложение, которое автоматизирует процесс приема, распределения, выполнения и контроля заявок на техническое обслуживание и поддержку.

- **Интерфейсы системы**: Веб-браузер (Google Chrome, Mozilla Firefox, Safari последних стабильных версий).
- **Интерфейсы пользователя**:
  - **Публичный портал для клиентов**: Веб-форма для создания заявок и просмотра их статуса.
  - **Интерфейс диспетчера**: Панель управления для просмотра заявок из всех источников, их классификации и назначения на группы.
  - **Интерфейс руководителя группы**: Просмотр всех заявок, назначенных на группу; распределение заявок по конкретным исполнителям; контроль загрузки исполнителей; просмотр сроков исполнения (SLA).
  - **Интерфейс исполнителя**: Список заявок, назначенных непосредственно на исполнителя руководителем группы, с возможностью изменения статуса заявки и добавления комментариев.
- **Интерфейсы аппаратных средств ЭВМ**: Стандартные серверные и клиентские аппаратные платформы.
- **Интерфейсы программного обеспечения**:
  - **Сервер электронной почты (SMTP)**: Для приема заявок по email и рассылки уведомлений клиентам.
  - **Телефонная линия (АТС)**: *[Опционально]* для автоматического создания заявки из звонка.
- **Интерфейсы коммуникаций**: Протокол HTTPS для безопасного доступа к веб-интерфейсу.
- **Ограничения памяти**: Система должна быть рассчитана на хранение данных о 10 000+ заявок.
- **Действия**:
  - Система должна обеспечивать круглосуточную доступность (24/7) за исключением плановых технических работ.
  - Система должна автоматически архивировать старые заявки по истечении заданного срока.

## Функции изделия
1. **Консолидация заявок:** Прием заявок через три канала (веб-форма, email, телефонный звонок) в единую систему.

2. **Управление справочной информацией:** Ведение справочников клиентов, групп сотрудников, типов проблем.

3. **Распределение заявок:** Функционал для диспетчера по анализу заявки и назначению ее на соответствующую группу исполнителей. Диспетчер не назначает заявки на конкретных сотрудников.
Ответственность за распределение внутри группы возлагается на руководителя группы. Также руководитель контролирует соблюдение сроков исполнения.

4. **Управление "workflow" (жизненным циклом) заявки:** Предоставление исполнителям возможности брать заявки в работу, помечать их как выполненные.

5. **Оповещение клиентов:** Автоматическая отправка email-уведомлений клиентам при изменении статуса заявки (принята в работу, выполнена).

6. **Обратная связь от клиента:** Механизм для клиента, позволяющий вернуть заявку в работу, если результат его не устроил.

7. **Управление SLA:** Назначение времени исполнения в зависимости от типа заявки и отображение срока исполнения (дедлайна) заявки исполнителю и руководителю.

8. **Распределение заявок руководителем группы:** Руководитель видит все активные заявки своей группы и назначает заявку на конкретного исполнителя.

9. **Мониторинг просроченных заявок:** Просмотр списка просроченных заявок и просмотр исполнителей, которые не успели выполнить заявки в срок.

## Характеристики пользователей
| Роль       | Описание | Уровень подготовки         |
|------------|--------|------------------|
| Клиент | Сотрудник предприятия-заказчика    | Базовые навыки работы с веб-браузером и электронной почтой |
| Диспетчер | Сотрудник компании, ответственный за координацию    | Высокий уровень знаний о структуре компании, группах и типах проблем. Уверенный пользователь ПК. |
| Руководитель группы | Сотрудник компании, отвечающий за распределение заявок внутри своей группы, контроль загрузки исполнителей и соблюдения сроков исполнения (SLA) | Уверенный пользователь ПК. Глубокое понимание структуры своей группы и типов решаемых проблем. |
| Исполнитель | Технический специалист (инженер, программист)    | Уверенный пользователь ПК. Специализированные технические знания в своей области. |
| Администратор системы | Сотрудник, отвечающий за настройку системы    | Глубокие знания системы, умение работать со справочниками и настройками |

## Ограничения
*   **Организационные:**
    *   Все заявки, независимо от источника поступления, в обязательном порядке проходят через диспетчера для первичной обработки и классификации.
    *   Диспетчер может назначить заявку только на группу. Назначать заявку на конкретного исполнителя может только руководитель группы.
    * Руководитель группы обладает компетенцией и правами для распределения заявок между исполнителями и оценки их загрузки.
    *   Возвратить заявку на доработку имеет право только клиент, от имени которого была создана заявка.

*   **Технические:**
    *   Система реализована как веб-приложение.
    *   Интерфейс системы поддерживает только русский язык.

*   **Эксплуатационные:**
    *   Резервное копирование данных выполняется один раз в 24 часа.
    *   Плановые технические работы проводятся в нерабочее время и не должны превышать 4 часов в месяц.

## Предположения и зависимости
*   **Предположения:**
    *   Диспетчер обладает достаточной квалификацией для корректного определения типа проблемы и назначения заявки на нужную группу.
    *   Каждый клиент имеет действующий и доступный адрес электронной почты и регулярно его проверяет.
    *   Справочники (клиентов, групп, типов проблем) поддерживаются администратором системы в актуальном состоянии.

*   **Зависимости:**
    *   Работа системы зависит от стабильности интернет-соединения у всех пользователей.
    *   Функциональность приема заявок по email и отправки уведомлений зависит от доступности SMTP-сервера.
    *   Корректное отображение веб-интерфейса зависит от использования актуальных версий браузеров.

# 3. Детальные требования

Этот раздел детализирует требования к системе, включая функциональные и нефункциональные аспекты, необходимые для успешной реализации проекта.
Перед перечислением функциональных требований ниже приведено описание атрибутов (свойств), используемых для каждого требования.

## Атрибуты функциональных требований

Каждое функциональное требование включает следующие атрибуты:

### **Идентификатор**
Уникальный код, который однозначно идентифицирует требование во всей документации проекта.  
Структура: **FR-XX-NN**, где:
- **FR** - функциональное требование;
- **XX** - код роли (CUS – Клиент, DSP – Диспетчер, EXE – Исполнитель, ADM – Администратор, SYS – Система);
- **NN** - порядковый номер требования.

### **Название**
Краткое описание сути требования.

### **Описание**
Полное определение функциональности: что должно происходить, кто является инициатором действия, какие ключевые правила задействованы.

### **Приоритет**
Оценивает значимость требования:
- **Высокий** - критически важно, необходимо для базовой работы системы;
- **Средний** - важно, но может быть реализовано позже без критического ущерба;
- **Низкий** - желательно, улучшает удобство или расширяет функциональность.

### **Стабильность**
Оценка вероятности изменений:
- **Высокая** - требование точно не изменится;
- **Средняя** - возможны уточнения;
- **Низкая** - требование ещё обсуждается.

### **Важность**
Оценка влияния на бизнес-цели:
- **Высокая** - напрямую влияет на выполнение ключевых процессов;
- **Средняя** - влияет на эффективность работы;
- **Низкая** - улучшает UX или снижает нагрузку, но не критично.

## Производительность

Требования к производительности определяют скорость реакции системы при работе с заявками, а также допустимые уровни нагрузки при одновременной активности пользователей.

### Время отклика
- **Загрузка списка заявок** (клиент, диспетчер, исполнитель):  
  Не более **2 секунд** при объёме данных до 10 000 заявок.
- **Загрузка карточки заявки** (детальная информация, история, комментарии):  
  Не более **1 секунды**.
- **Создание заявки** (веб-портал):  
  Не более **700 мс** до подтверждения получения.
- **Создание заявки диспетчером** (по телефону):  
  Не более **700 мс**.
- **Обновление статуса заявки** (взята в работу, выполнена, возвращена):  
  Не более **500 мс**.
- **Автоматическое обновление интерфейса исполнителя** (push/refresh):  
  Новые заявки должны появляться в интерфейсе группы **в течение 3 секунд** после назначения.

### Пропускная способность
- Система должна выдерживать нагрузку **до 200 одновременных активных пользователей**, в том числе:
  - до **100 клиентов**, создающих и просматривающих заявки;
  - до **20 диспетчеров**, работающих в системе;
  - до **80 исполнителей**, обновляющих статусы и просматривающих списки.
- При этом не допускается деградация ключевых операций (создание/обновление заявки).

### Утилизация ресурсов
При пиковой нагрузке (200 одновременных пользователей):
- **Утилизация CPU сервера приложения** - не более **75%**.
- **Утилизация RAM сервера приложения** - не более **80%**.
- **База данных**:
  - время ответа на простой запрос (SELECT по ID заявки) - ≤ **50 мс**;
  - утилизация CPU - ≤ **70%**;
  - кэш-хитрейт БД - ≥ **85%**.
- Система должна масштабироваться горизонтально (приложение) без остановки работы.

### Обработка email-заявок
- Импорт писем с сервера должен происходить **не реже 1 раза в 10 секунд**.
- Создание заявки из email - не более **1 секунды** после получения письма системой.

### Уведомления клиентам
- Email-уведомление о смене статуса должно отправляться **в течение 5 секунд** после изменения статуса заявки.

## Ремонтопригодность
Требования к ремонтопригодности определяют простоту сопровождения и модернизации системы.

### Архитектура и код
- Система должна иметь модульную структуру, позволяющую изменять отдельные компоненты без затрагивания всей системы.
- Код должен быть документирован и соответствовать единым правилам оформления.
- Настройки системы (БД, SMTP, пути интеграции) должны храниться в конфигурационных файлах, отдельно от кода.

### Обновление и поддержка
- Обновления должны выполняться с минимальным простоем (не более 5 минут).
- Должна быть возможность отката версии (rollback) при неуспешном обновлении.
- Система должна предоставлять диагностическую информацию через логи.

### Тестирование
- Ключевые функции (создание, обновление статусов, уведомления) должны быть покрыты базовыми автотестами.
- Перед установкой новой версии должна выполняться регрессионная проверка основных сценариев.

## Ограничения проекта

Ограничения определяют рамки, в пределах которых допускается принимать архитектурные, технические и организационные решения при разработке системы.

### Ограничения инфраструктуры
- Система должна работать в инфраструктуре заказчика - использование внешних облачных платформ (AWS/Azure/GCP) не допускается без отдельного согласования.
- СУБД - PostgreSQL (или другая, доступная в инфраструктуре заказчика). Переход на новые СУБД или кластерные решения не допускается.
- Интеграция с email может выполняться **только через существующие корпоративные SMTP/IMAP/POP3-сервера**.
- Доступные серверные мощности определяются заказчиком - масштабирование ограничено текущими ресурсами.

### Ограничения по стеку технологий
- Разрешённые серверные стеки: **Java (Spring Boot)** или **.NET Core**.  
  Использование других языков и фреймворков допускается только при письменном согласовании.
- Фронтенд - только веб-интерфейс; мобильные приложения не предусмотрены.
- Интерфейс системы - **только на русском языке**, мультиязычность не требуется.
- Все коммуникации должны работать по HTTPS и использовать UTF-8 - альтернативные протоколы или нестандартные кодировки не допускаются.

### Ограничения по интеграциям
- Интеграция с корпоративными каталогами возможна только через LDAP/LDAPS (версии протокола менять нельзя).
- Интеграция с другими системами допускается **только через REST API**; использование SOAP или проприетарных протоколов исключено.
- Отправка уведомлений возможна исключительно через корпоративный почтовый шлюз (сторонние облачные сервисы запрещены).

### Ограничения безопасности
- Все решения должны соответствовать внутренним политикам ИБ заказчика (дополнительные библиотеки, хранилища и сторонние сервисы проходят обязательное согласование).
- Данные и логи должны храниться только во внутренней сети; внешние лог-хранилища (SaaS ELK/CloudWatch и т.д.) запрещены.
- Используемые компоненты должны иметь открытые и совместимые лицензии (MIT, Apache 2.0, BSD).  
  Компоненты с GPL и аналогичными ограничивающими лицензиями использовать нельзя.

### Ограничения эксплуатации и сопровождения
- Резервное копирование данных выполняется раз в 24 часа - проект обязан учитывать этот цикл (например, при восстановлении или валидации данных).
- Плановые работы ограничены суммарно 4 часами в месяц - архитектура должна минимизировать необходимость простоя.
- Поддержка устаревших браузеров (например, Internet Explorer) отсутствует - интерфейс разрабатывается только под современные версии Chrome, Firefox, Edge.

### Организационные ограничения (влияющие на проект)
- Все заявки проходят через диспетчера - архитектура должна предусматривать соответствующие роли и маршруты.
- Заявки назначаются только на группы сотрудников - индивидуальное назначение не реализуется.
- Возврат заявки на доработку выполняется только клиентом - логика статусов и переходов должна учитывать это ограничение.
- Пользователи не проходят длительного обучения - интерфейс должен быть максимально простым; сложные интерфейсы и многоуровневые меню не допускаются.
- Процессы внедрения и изменений должны соответствовать корпоративным регламентам - быстрые экспериментальные развёртывания и частые релизы невозможны.

### Ограничения по данным
- Срок хранения заявок минимум 3 года - архитектура должна учитывать объём хранимых данных и нагрузку на СУБД.
- Хранилище вложений ограничено ресурсами заказчика - использование внешних файловых сервисов (Google Drive, S3) запрещено.
- Система должна корректно обрабатывать большие объёмы входящих писем - но без возможности использовать очередь сообщений вне инфраструктуры заказчика.

### Организационные ограничения
- Все заявки, независимо от источника, в обязательном порядке проходят через диспетчера для первичной классификации.
- Назначение заявки выполняется **только на группу сотрудников**, а не на конкретного исполнителя.
- Возврат заявки на доработку может осуществлять только клиент, от имени которого она создана.
- Все изменения в системе должны соответствовать корпоративным процессам управления изменениями и политике безопасности заказчика.
- Пользователи не проходят длительного обучения, поэтому интерфейс должен быть максимально простым и интуитивным.

### Технические ограничения

- Интерфейс поддерживает только русский язык.
- Вся логика работы с почтой должна использовать стандартные протоколы IMAP/POP3/SMTP.
