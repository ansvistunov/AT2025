# 1. Введение
## Цели

Настоящая спецификация требований программного обеспечения (SRS) описывает требования к разработке системы **Приложение для проката самокатов**.  
Цель документа — определить функциональные, эксплуатационные и интерфейсные требования к программному обеспечению, обеспечивающему контроль и геолокацию самокатов с помощю встроенных модулей, пользовательский интерфейс взаимодействия с самокатом. 
Документ предназначен для:
- заказчика (ПАО "МТС");
- разработчика системы и программного обеспечения;
- специалистов, ответственных за эксплуатацию, техническое обслуживание и внедрение в экосистему МТС.


## Границы применения
Приложение имеет две версии.
1. Пользовательская - предназначено для мониторинга геолокации самоката, а так же отслеживания заряда батареи и стоимость подачи. Его основная функция — обеспечение взаимодействя клиентов с сервисом проката.
2. Сервисная - преднозначено для мониторинга геолокации самокатов, отслеживание технической информации(Пробег, состояние батареи, коды ошибок, флаги блокировки). Его основная функция - обеспечение своевременного технического обслуживания и противоугонной системы.

# 2. Общее описание

## Описание изделия
- **Интерфейсы системы** Для работы приложения тербуется Android 8+, Java библиотеки, Мобильный ID(МТС), Android Location services, PostgreSQL, RxJava.
- **Интерфейсы пользователя** Интерфейс пользователя состоит из:
1. Экран Авторизации;
2. Карта геолокации
3. Личный кабинет
4. Поддержка
5. Настройки
- **Интерфейсы коммуникаций**Интерфес чата с поддержкой, боты помощи по ЧаВО, 
- **Ограничения памяти** От 3гб ОЗУ, от 500+мб без учета кеша приложения.
- **Действия**Сценарий пользователя:
Авторизация в приложеинии-> Обнаружение совбодного самоката на карте-> Нахождение самоката пользователем на улице-> Сакнирование QR кода-> Списание залога(установленного заказчиком)-> Старт сессии поездки-> Поездка-> Завершение поездки пользователем через приложение-> Списание платы с учетом разницы залога.
Сценарий технического персонала:
Авторизация в приложеинии-> Обнаружение самоката тербуещего обслуживания-> Блокировка самоката до прибытия специалиста-> Действия с самокатом технического характера-> Снятие блокировки-> Самокат возвращен в работу.
## Функции изделия
1. Автоматизация услуг проката самокатов
2. Система отслеживания геолокации самокатов
3. Противоугонная система

## Ограничения
1. Регистрация пользователей только через МТС ID
2. Отсутствие глобального ограничения на максимальное количество пользователей
3. Минимальное количество участников устанавливается системой и не может быть изменено пользователями
4. Оплата взымается только по предварительно зарегестрированному счету
## Предположения и зависимости
1. Все пользователи имеют стабильный доступ к приложению.
2. В случае парковки самоката в неустановленном месте с пользователя взымается штраф.
3. Система использует платежные шлюзы установленные заказчиком.
4. Модуль геолокации работает максимально безотказно до внешнего вмешательсва со стороны недобростоветсных пользователей.

# 3. Детальные требования
В данном разделе описываются функциональные и нефункциональные требования к системе 
## Функциональные требования
Функциональные требования определяют, что должна делать система. Они сгруппированы по ролям пользователей и системным функциям.

* **Идентификатор**
    * Уникальный код, который идентифицирует требование во всей документации проекта. Он позволяет ссылаться на конкретное требование без двусмысленности.
    * Структура: `FR-XX-NN`, где `FR` = Функциональное требование, `XX` = код роли (TL, TR, HR и т.д.), `NN` = порядковый номер.
* **Название**
    * Краткое, но емкое наименование функции, описывающее ее основную суть.
* **Описание**
    * Полное определение требования. Разъясняет, что система должна делать, кто является пользователем, и какие основные шаги или правила вовлечены.
* **Приоритет**
    * Определяет важность требования для проекта и его критичность для основной функциональности. Этот атрибут используется для определения порядка реализации.
    * **Высокий:** Критически важное требование. Без его реализации система не сможет выполнять свои основные бизнес-задачи. Должно быть реализовано в первой версии.
    * **Средний:** Важное требование. Влияет на ключевые, но не критические функции. Система может работать с ограничениями, но основной бизнес-процесс будет нарушен или неудобен.
    * **Низкий:** Желательное требование. Касается второстепенных функций, удобства использования или расширенной отчетности. Его отсутствие не блокирует основной функционал.

## Надежность
- **Доступность**: 100% в пределах зоны покрытия, а так же во время пользования системой клиентом, система работет 24 часа в сутки без остановок.
- **MTBF**: Среднее время непрерывной работы системы между двумя критическими сбоями должно составлять не менее 1 000 часов..
- **MTTR**: 1 час. Критические сбои, делающие систему полностью недоступной, должны быть устранены в течение 1 часа в рабочее время.
- **Точность**: Система должна обеспечивать 95% точность трекинга самокатов, оплата должна производиться без задержек по точным реквезитам.
- **Макс. количество ошибок**: Не более 3 "Критических" или "Высоких" дефектов, блокирующих основной функционал, на момент релиза.

## Производительность
Требования к производительности определяют, насколько быстро система реагирует на действия пользователя при определенной нагрузке.
- **Время отклика**:
    *Загрузка страниц и отображение списков (заявки, сессии): не более 2 секунд.
    *Выполнение транзакций (создание заявки, бронирование, отправка отзыва): не более **500 мс**.
- **Пропускная способность**:  Система должна поддерживать одновременную работу до **100 000** активных пользователей без деградации производительности.
- **Утилизация ресурсов**: При пиковой нагрузке (100 000 одновременных пользователей) утилизация CPU и RAM сервера приложения не должна превышать **80%**.

## Ремонтопригодность
Требования к ремонтопригодности определяют легкость, с которой система может быть модифицирована или исправлена.
* Модули геолокации должны быть взаимозаменяемы и не привязаны к серийному номеру самоката
* В случае поломки или технических работ на сервере для обработки геолокации необходимо использовать резервные серверные мощности или ограничивать доступность ситемы для пользователей на время тхнических работ
* Процесс развертывания (деплоя) исправлений (hotfix) на рабочем сервере не должен занимать более **90 минут** и не должен требовать полной остановки системы.

## Ограничения проекта
Ограничения проекта — это факторы, которые ограничивают выбор решений при разработке.
* **Технологический стек:** Продукт должен быть разработан с использованием текущего технологического стека компании (например, Java библиотеки).
* **Аутентификация (SSO):** Аутентификация пользователей должна осуществляться через **MTS ID**.
* **Среда:** Система должна быть андроид-приложением.
* **Бюджет и сроки:** Проект должен быть завершен в течение 6 месяцев с командой из 6 разработчиков.
* **Данные:** Система должна хранить платежные данные клиента, и должна соответствовать внутренним политикам безопасности данных компании.
## Требования к пользовательской документации
Система должна быть сопровождена следующей пользовательской документацией, предназначенной для различных категорий пользователей:

* **Руководство пользователя:**
**Формат** : Онлайн-справка, интегрированная в андроид- приложение (контекстно-зависимые подсказки), а также PDF-документ.
**Содержание**: Должно включать пошаговые инструкции с иллюстрациями (скриншотами) по выполнению всех ключевых сценариев для каждой роли: создание заявки, подача предложения, бронирование, запись на занятие, оставление отзывов и оценок.
**Язык**: Русский. Текст должен быть понятен пользователям с базовым уровнем компьютерной грамотности.

* Запрещается использование компонентов с лицензиями, обязывающими раскрытие исходного кода всей системы (например, GPL), без явного одобрения юридическим отделом компании. Предпочтение отдается лицензиям типа MIT, Apache 2.0, BSD.
